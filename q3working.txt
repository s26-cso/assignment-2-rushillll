i started the same way and did an obj dump. the main function looked different and instead of scanf it was calling _IO_gets which is gets(). gets has no bound checking defined here and itll take wtv input i give it.

also, after gets returns, the code does mv a0, sp which sets a0 to the stack pointer, and then immediately does bnez a0 which branches to .fail if a0 is not zero. since the sp is never zero, this means the .pass block is completely unreachable through normal execution. itll always print the sorry, and you cant reach pass via input

i tried giving a very large input and the program crashed with a seg fault. this confirmed that gets was overflowing the buffer and corrupting the stack.

the buffer allocated for input was 112 bytes and above that on the stack was 8 bytes for a saved register, and then 8 bytes for the saved ra. so, the return address sits at offset 120 from the start of the buffer. when main finishes and executes ret, it jumps to whatever address is stored there.

Since the binary is compiled with -no-pie, the address of .pass is fixed and i could see it directly in the objdump output as 0x104e8. so, i can send 120 bytes of junk to fill up the buffer and the saved register, and then the 8 byte address of .pass in little endian format. wheb main returns itll jump to .pass instead of back to the standard library, and .pass prints what i need
